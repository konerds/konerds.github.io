Java 언어, JVM 및 생태계

1990년대 후반부터 등장한 Java 프로그래밍 환경은 크게 Java 언어와 이에 대한 런타임 환경을 제공하는 JVM(Java Virtual Machine) 으로 구성됩니다. Java가 처음 개발되었을 당시 이러한 분할된 구조의 환경은 신박하다는 평가를 받았으며, Java 가 등장하고 몇 년 후에 발표된 Microsoft 의 .NET 환경도 매우 유사한 플랫폼 아키텍처 접근 방식을 채택했습니다. Microsoft 의 .NET 플랫폼과 Java 의 중요한 차이점 중 하나는 Java 는 항상 여러 공급업체를 통해 비교적 개방적인 생태계로 발전되었다는 점입니다.

Java 가 성공할 수 있는 주요 이유 중 하나로 이런 개방적인 생태계가 표준화된 환경이라는 점을 꼽을 수 있습니다. 이는 환경을 구성하는 기술 사양이 있음을 의미합니다. 이러한 표준은 서로 다른 공급업체를 통해 구현된 제품이라 할지라도 다른 구성 요소와 호환될 것이라는 확신을 줍니다.

Oracle 이 Sun Microsystems 를 인수한 이래로 Java 는 Oracle 에서 소유하고 있습니다. Red Hat, IBM, Hewlett-Packard, SAP, Apple, Fujitsu 와 같은 다른 기업들도 표준화된 Java 기술 사양에 깊이 관여하고 있습니다. OpenJDK 라는 오픈 소스 버전의 Java도 있으며, 앞서 언급한 회사들 중 다수가 협력하고 있습니다.

Java 는 실제로 Java ME(Java Micro Edition), Java SE(Java Standard Edition), Java EE(Java Enterprise Edition) 등 서로 다르지만 관련된 환경과 사양으로 구성됩니다.

본 섹션에서는 Java SE 8 을 대상으로 포스팅을 진행할 예정입니다.


Java 언어

Java 프로그램은 Java 언어의 소스 코드로 작성됩니다. 이는 사람이 읽을 수 있는 클래스 기반의 객체 지향 프로그래밍 언어입니다. 때때로 약간 장황하더라도 읽고 쓰기가 상대적으로 쉬운 것으로 간주됩니다.

Java 언어는 배우고 가르치기 쉽도록 만들어졌습니다. C/C++ 같은 초기 프로그래밍 언어에 대한 업계 경험을 바탕으로 복잡한 기능들을 제거하고 이전 프로그래밍 언어에서의 동작들을 최대한 보존하며 설계되었습니다.

전반적으로 Java 는 기업이 비즈니스에 중요한 프로그램을 개발할 수 있는 안정적이고 견고한 기반을 제공합니다. 따라서 상대적으로 보수적인 설계 구조와 더딘 발전 속도를 유지합니다.

Java 언어는 1996년 처음 출시된 이후, 점진적인 개정을 통해 발전해왔습니다. 이는 1990년대 후반에 선택된 초기 설계 구조의 골자가 오늘날까지도 여전히 영향을 미치고 있음을 의미합니다.

이런 더딘 발전 속도에도 불구하고, Java 8 은 거의 10년 동안 볼 수 있었던 가장 급진적인 행보를 보여줍니다. 람다 표현식 및 핵심 컬렉션, 코드 정밀 검사 등 대부분의 Java 개발자가 코드를 작성하는 방식에 대한 패러다임에 변화를 가져왔습니다.

Java 언어 구현에 대한 작동 원리는 JLS(Java Language Specification) 에 의해 정의 및 관리되고 있습니다.


JVM(Java Virtual Machine)

JVM 은 Java 프로그램을 실행하는데 필요한 런타임 환경을 제공하는 프로그램입니다.

하드웨어 및 OS 플랫폼에 호환되는 JVM 이 제공되어야만 Java 프로그램을 실행할 수 있습니다. 다행스럽게도 JVM 은 셋톱박스나 블루레이 플레이어부터 거대한 메인 프레임에 이르기까지 다양한 환경에서 실행되도록 포팅되어 있어서 대부분의 환경에서 호환됩니다.

Java 프로그램은 일반적으로 다음과 같은 명령줄로 시작됩니다.

java <arguments> <program name>

이는 Java 런타임 환경을 제공하는 운영 체제 프로세스로, JVM 을 불러온 다음 새로 시작된 빈 가상 머신의 컨텍스트에서 프로그램을 실행합니다.

JVM 이 실행을 위해 Java 프로그램을 읽을 때 이는 Java 소스 코드로 제공되지 않는다는 점을 이해하는 것이 중요합니다. 대신, Java 소스 코드는 사전에 Java 컴파일러를 통해 Java 바이트 코드라는 형식으로 변화되어야 합니다. Java 바이트코드는 항상 .class 파일 형식으로 JVM 에 제공되어야 합니다.

JVM 은 프로그램의 바이트 코드 형식을 위한 인터프리터로서 한 번에 하나의 바이트 코드 명령을 실행합니다. 그러나 JVM 과 작성된 프로그램 모두 추가적인 실행 스레드를 생성할 수 있으므로, 작성된 프로그램이 동시에 실행되는 다양한 기능을 가질 수 있다는 점도 알아야 합니다.

JVM 의 설계는 C/C++ 언어를 비롯한 초기 프로그래밍 언어 환경에 대해 축적된 경험을 바탕으로 구축되었으므로, 프로그래머의 삶을 더 쉽게 만들기 위한 여러 가지 목표를 가지고 있습니다.

  • 내부에서 프로그램 코드를 실행할 수 있도록 구성된 컨테이너 제공
  • C/C++ 언어 대비 안전한 실행 환경 제공
  • 메모리 관리 작업 자동 수행
  • 크로스 플랫폼 실행 환경 제공

이러한 목표는 플랫폼을 논의할 때 종종 함께 언급됩니다.

앞서 JVM 과 해당 바이트 코드 인터프리터에 대해 논의할 때 이러한 목표 중 첫 번째 목표를 이미 언급했습니다. JVM 은 애플리케이션 코드의 컨테이너 역할을 합니다.

WORA(Write Once, Run Anywhere) 라고도 불리는 네 번째 목표는 JVM 만 설치되어 있으면 하나의 Java 소스 코드로 컴파일된 Java 바이트 코드 파일을 서로 다른 플랫폼에서 실행할 수 있음을 의미합니다.

Java 환경은 Mac, Windows, Linux, Solaris, BSD Unix, AIX 등 다양한 플랫폼에서 실행될 수 있습니다

JVM 은 이러한 네 가지 기본 목표 외에 자체 관리를 위해 런타임 정보를 사용할 수 있도록 설계되었다는 점에도 주목할 필요가 있습니다.

1970년대와 1980년대의 소프트웨어 연구에 따르면 프로그램의 런타임 동작에는 컴파일 타임에 추론될 수 없는 흥미롭고 유용한 패턴이 많이 있다는 것이 밝혀졌는데, 이 연구를 활용하여 설계된 최초의 플랫폼이 바로 JVM 입니다. JVM 은 보다 효율적으로 코드를 실행하기 위해 런타임 정보를 수집하는데, 이는 실행 중인 프로그램을 모니터링하고 최적화할 수 있음을 의미합니다.

중요한 예시를 들면 Java 프로그램의 부분마다 프로그램 수명 동안 호출되는 빈도가 서로 다르다는 사실을 들 수 있습니다. JVM 은 다른 부분보다 비교적 훨씬 자주 호출되는 코드들이 있다면, JIT(Just-In-Time) 컴파일 기술을 통해 이러한 비효율적인 호출을 방지합니다. Sun Microsystems 에서 처음 Java 1.3 일부로 제공한 이래로 오늘날까지도 사용되고 있는 HotSpot JVM 은 먼저 프로그램의 어느 부분이 가장 자주 호출되는지 식별하여 이를 핫 메소드로 저장합니다. 그런 다음 JVM 은 이러한 핫 메소드에 해당되는 코드를 만나면 JVM 인터프리터를 우회하여 사전에 저장된 머신 코드를 직접 읽습니다. 따라서 JVM 인터프리터를 통해 순수하게 실행되는 경우보다 높은 성능을 제공합니다. 실제로 JVM 이 현재 사용하는 이러한 최적화 과정을 통해 C/C++ 컴파일러와 가까운 성능을 보여주기도 합니다.

JVM 의 동작 방식에 대한 표준은 JVM Specification 에 의해 정의 및 관리되고 있습니다.


JAVA 생태계

Java 언어는 배우기 쉽고 다른 프로그래밍 언어에 비해 추상화가 상대적으로 적습니다. 더불어 JVM 은 Java 프로그램을 실행할 수 있는 견고하고 이식 가능한 고성능 기반을 제공합니다.

Java 언어와 JVM 은 이러한 장점들을 통해 기업이 어느 부분을 중점적으로 개발할 지에 대한 의사 결정에 있어서 확신을 가질 수 있는 기반을 제공합니다.

그러나 Java 의 장점은 여기서 끝나지 않습니다. Java 가 시작된 이래로 타사 라이브러리 및 구성 요소로 구성된 매우 큰 생태계가 성장했습니다. 따라서 현대 프로그래밍 기술 생태계에서 Java 커넥터를 제공하지 않는 경우는 실제로 거의 드뭅니다.

오픈 소스 여부와 관계 없이 대부분의 기술에 대한 커넥터와 드라이버가 이미 존재하고 있습니다. 전통적인 관계형 데이터베이스부터 NoSQL, 모든 유형의 기업 모니터링 시스템, 메시징 시스템까지 다양한 구성 요소들이 Java 를 지원합니다.

이것이 바로 많은 기업들이 Java 기술을 채택하게 된 주요 원인입니다. Java 를 통해 개발자들은 기존 라이브러리와 구성 요소를 활용하여 잠재력을 발휘할 수 있습니다.
